# GSD Wallet

Chrome extension wallet for dApp developers on the Midnight blockchain. **This is a development tool, not a production wallet.** Seeds are stored unencrypted. DApp connections and transactions are auto-approved.

## Architecture

```
Page (DApp) <-> Content Script <-> Service Worker (router) <-> Offscreen (relay) <-> Web Worker (SDK host)
```

| Component | Location | Role |
|-----------|----------|------|
| Service worker | `src/background/` | Message routing, session management, state caching, diagnostics |
| Offscreen relay | `src/offscreen/offscreen.ts` | Thin port-to-postMessage bridge (635B) |
| Web Worker | `src/offscreen/worker.ts` | SDK host — WalletFacade, sync, TX processing, GSD Connect |
| Popup UI | `src/popup/` | React 19, Zustand, Tailwind |
| Content script | `src/content-script/` | DApp connector bridge (`inpage.js` injected into page) |
| Shared | `src/shared/` | Types, constants, environments, storage, messages |
| Core | `src/core/` | Transfer, dust registration, balance formatting, address validation |

The offscreen document is persistent (Chrome does not GC it). The Web Worker hosts all SDK state and survives SW restarts. The offscreen relay reconnects to the SW with exponential backoff (500ms-30s).

## Domain

### Tokens and resources

- **NIGHT** — Native token. Smallest unit: **star** (1 NIGHT = 10^6 stars)
- **DUST** — On-chain resource (not a token). Generated by registered NIGHT UTXOs, consumed as transaction fees. Cannot be transferred. Smallest unit: **speck** (1 DUST = 10^15 specks)

### Three-wallet model

The SDK syncs three subsystems independently:

| Subsystem | Privacy model | Why |
|-----------|--------------|-----|
| Shielded | All events replayed locally | ZK privacy — indexer never sees viewing keys |
| Unshielded | Filtered by address | Public — indexer can filter without privacy risk |
| Dust | All events replayed locally | Same privacy model as shielded |

### Transaction flow

All transactions serialize through a single queue (`withTxQueue` in `worker.ts`). The flow is: `transferTransaction` -> `signRecipe` -> `finalizeRecipe` -> `submitTransaction`. TTL is 30 minutes (`TX_TTL_MS`).

## SDK dependencies

The wallet builds on `@midnight-ntwrk/wallet-sdk-facade` which wraps shielded, unshielded, and dust wallets. The facade is initialized via `WalletFacade.init()` (constructor is private).

**Do not maintain local SDK reference docs.** The authoritative source is the [midnight-wallet](https://github.com/midnightntwrk/midnight-wallet) repo. See `docs/sdk-reference/README.md` for links.

## Key conventions

- **DApp connector methods**: 17 methods in `inpage.js` / `connectedApiHandler.ts`. The method list must match in both files.
- **Diagnostic events**: `emit(level, category, message, data?, elapsed?)` — categories include sw, wallet, state, sync, api, dapp, tx, connect, error, storage
- **State broadcasts**: Worker -> offscreen relay -> SW -> popup ports + session storage cache
- **GSD Connect**: WebSocket client in Worker (`connectClient.ts`) connects to external Node.js server. DApp requests route through `handleDappRequest()` in the SW, sharing session tracking with browser dApps.

## Proving

Transactions require zero-knowledge proofs. Two modes are available, controlled by a `ProvingStrategy` with a k-value threshold:

| Strategy | kThreshold | Behavior |
|----------|-----------|----------|
| WASM (all) | `Infinity` | All proofs via in-browser WASM. No server needed. |
| WASM <=N | `10`-`17` | WASM for circuits with k<=N, proof server for k>N |
| Server only | `0` | All proofs via external proof server at `localhost:6300` |

### Architecture

```
WalletFacade.finalizeRecipe()
  -> compositeProvingService.prove(tx)
    -> trackingKeyMaterialProvider.getParams(k)
      -> if k > kThreshold: throw KValueExceededError -> fall back to server
      -> if aborted: throw ProvingCancelledError -> fall back to server
    -> makeWasmProvingService / makeServerProvingService
```

The composite service (`src/offscreen/compositeProvingService.ts`) wraps both WASM and server provers. It creates a tracking `KeyMaterialProvider` wrapper that intercepts `getParams(k)` to check the threshold before loading BLS params. Cancellation uses `AbortController` — the WASM Worker is terminated cleanly.

### Key material

Key material is resolved via a three-tier cache in `src/offscreen/keyMaterialProvider.ts`:
1. In-memory Map (hot path)
2. IndexedDB stores (`provingKeys`, `provingParams`)
3. Bundled files via `chrome.runtime.getURL('data/proving/...')` (k<=16)
4. S3 fallback for unbundled k values (k>=17)

Bundled files are fetched via `scripts/fetch-proving-data.sh` (not committed to git). Circuit keys use version 9 paths (`zswap/9/spend.prover` etc.).

### Device benchmark

On first install or on demand, a k=10 proof fixture is proved via WASM. The wall-clock time is extrapolated using the doubling model: `time(k) = k10Time * 2^(k-10)`. Results are stored in `chrome.storage.local` as `gsdDeviceBenchmark` and shown in the header dropdown.

### Strategy persistence

The active strategy is stored in `chrome.storage.local` as `gsdProvingStrategy` and restored on extension restart. Default: `{ kThreshold: 17 }`.
